# Layouts Reference

App-shell templates generated by `/app-layout`.

## Shells

| Shell | When to use | Grid areas |
|---|---|---|
| `holy-grail` | Content-heavy apps with primary + secondary sidebars (docs, admin) | `"header header header" / "nav main aside" / "footer footer footer"` |
| `sidebar` | Typical SaaS dashboards (one fixed sidebar + content) | `"nav header" / "nav main" / "nav footer"` |
| `centered` | Auth screens, 404/500 pages, onboarding | single centered column, `max-inline-size: 32rem` |

## Responsive behavior

Breakpoints follow the fpkit convention (rem, mobile-first):

- `<= 48rem` (~768px): collapse to single column; nav becomes a toggleable off-canvas.
- `48rem–64rem`: sidebar fixed; aside hides.
- `> 64rem`: full three-column grid.

Grid areas, gaps, and breakpoints are exposed as CSS variables (see `assets/layouts/app-shell.scss`):

```scss
--app-shell-grid-template
--app-shell-gap
--app-shell-header-block-size
--app-shell-nav-inline-size
--app-shell-aside-inline-size
```

## Semantic landmarks

Every shell uses native landmarks for screen-reader navigation:

- `<header>` — top bar, branding
- `<nav>` — primary navigation (single per page)
- `<main>` — page content (single per page)
- `<aside>` — complementary content
- `<footer>` — page footer

When `source=generated`, shells wrap children in the polymorphic `UI` component so theme tokens cascade. When `source=npm`, native elements are used directly (the named `Header`/`Main`/etc. exports are not guaranteed — see `component-source.md`).

## Page slot composition

Pages consume the shell via children:

```tsx
import AppShell from './AppShell'
import Overview from '../pages/Overview'

export default function App() {
  return (
    <AppShell>
      <Overview />
    </AppShell>
  )
}
```

Pages may project content into named slots (`header`, `nav`, `aside`, `footer`) via render props — the shell's `header`, `nav`, `aside`, `footer` props accept React nodes.
