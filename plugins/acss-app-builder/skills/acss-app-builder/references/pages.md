## Settings — tab pattern clarification

`Details` is a disclosure widget (collapsing sections), not tabs. For true tab semantics, use this ARIA pattern:

```tsx
<nav role="tablist" aria-label="Settings sections">
  <button role="tab" aria-selected={active === 'profile'} aria-controls="panel-profile" id="tab-profile">Profile</button>
  <button role="tab" aria-selected={active === 'notifications'} aria-controls="panel-notifications" id="tab-notifications">Notifications</button>
</nav>
<div role="tabpanel" id="panel-profile" aria-labelledby="tab-profile" hidden={active !== 'profile'}>…</div>
```

Keyboard: Left/Right arrows move between tabs; Home/End jump to first/last. The `settings.tsx` template wires this.

## Page file conventions

- One page per file under `src/pages/`.
- Default export is the page component.
- Named export `<Name>Loader` (optional) for data-fetch hooks — not wired in v0.1.
- Each page imports its own SCSS if it needs page-scoped styling.

## Accessibility checklist (applied to every generated page)

- Single `<h1>` per page.
- All interactive elements reachable via keyboard.
- Visible focus indicators (inherited from fpkit component styles).
- Form fields use `Field` wrapper with `labelFor` (see `forms.md`).
- Tables include `<caption>` or `aria-label`.
- Dialogs trap focus (fpkit `Dialog` handles this).
