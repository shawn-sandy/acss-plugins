# Themes Reference

Theme presets generated by `/app-theme <preset>`.

## Presets

| Preset | Files produced | Strategy |
|---|---|---|
| `light` | `src/styles/theme/light.css` | Semantic roles → palette tokens. |
| `dark` | `src/styles/theme/dark.css` | Uses `[data-theme="dark"]` scope, not `@media (prefers-color-scheme)`. |
| `both` | `light.css` + `dark.css` | Pair above. Wire the developer's toggle to set `data-theme` on `<html>`. |
| `brand-neutral` | `src/styles/theme/brand-neutral.css` | Overrides brand accent tokens; stays palette-agnostic. |
| `brand-vibrant` | `src/styles/theme/brand-vibrant.css` | Saturated accent palette. |

All preset files are **hand-tuned** and committed verbatim from `assets/themes/`. v0.1 does not auto-generate palettes from a seed color.

## Token layers

Themes follow a three-layer cascade (authoritative spec at `docs/css-variables.md`):

1. **Primitives** (in `base.css`) — `--color-slate-900`, `--color-blue-500`, `--space-1`…
2. **Semantic roles** (in light/dark files) — `--color-background`, `--color-text`, `--color-primary`, `--color-surface`.
3. **Component tokens** (consumed by fpkit components) — `--btn-bg: var(--color-primary)`.

## Naming rules (theme tokens)

Theme files define three categories of tokens. The validator landscape differs by category:

| Category | Where | Example | Validator |
|---|---|---|---|
| Primitive design tokens | `base.css` | `--fs-md`, `--space-4`, `--radius-sm` | None — scales are exempt |
| Semantic role tokens | `light.css`, `dark.css`, `brand-*.css` | `--color-primary`, `--color-text` | `validate_theme.py` (contrast) |
| Component-scoped tokens | component SCSS files, not theme files | `--btn-bg`, `--card-header-padding` | `validate_css_vars.py` (naming pattern) |

Rules for all theme tokens:

- All values in **rem**, never **px**.
- Semantic role tokens use `--color-*`, `--space-*`, `--fs-*`, etc. — not `--{component}-{property}`.
- Component-scoped tokens in SCSS follow `--{component}-{element?}-{variant?}-{property}-{state?}` and are validated by `validate_css_vars.py`.
- Approved abbreviations: `bg`, `fs`, `fw`, `gap`, `radius`. Full words for `padding`, `margin`, `color`, `border`, `display`, `width`, `height`.
- Use logical properties: `padding-inline`, `padding-block`.

## Dark mode cascade

```css
/* light.css */
:root {
  --color-background: #ffffff;
  --color-text: #1a1a1a;
  --color-primary: #0066cc;
}

/* dark.css */
[data-theme="dark"] {
  --color-background: #1a1a1a;
  --color-text: #f5f5f5;
  --color-primary: #66b3ff;
}
```

Toggle by setting `<html data-theme="dark">`. This avoids the flash-of-wrong-theme that `prefers-color-scheme` alone produces when the user has overridden their OS preference.

## Contrast validation

`validate_theme.py` (v0.1) validates **palette files only** — files whose name matches `{light,dark,brand-*}.css`. It:

1. Resolves one level of `var()` references to find hex literals.
2. For each pair (`--color-text` vs `--color-background`, `--color-primary` vs `--color-text-inverse`), computes WCAG contrast.
3. Flags pairs below 4.5:1 (normal text) or 3:1 (large text / UI controls).

Component-level theme files that only chain through `var()` without terminal hex literals are skipped with a note.

## Wiring

`/app-theme` appends CSS imports to the sentinel-bounded block in `src/main.tsx`:

```tsx
// @acss-app-builder:begin theme imports
import './styles/theme/base.css'
import './styles/theme/light.css'
import './styles/theme/dark.css'
// @acss-app-builder:end
```
